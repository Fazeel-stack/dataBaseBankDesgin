# Relational Banking Database System

A SQL Server relational database that simulates core banking operations such as customer onboarding, account management, deposits, withdrawals, transfers, and audit logging. The project demonstrates how business rules can be enforced directly at the database layer using T-SQL, stored procedures, transactions, constraints, and triggers.


## Overview

This project was built to explore database design beyond basic CRUD operations by implementing business logic directly inside SQL Server.

Rather than allowing unrestricted table modifications, all core operations are performed through stored procedures. Multi-step workflows are executed inside transactions to maintain consistency, while triggers automatically record balance updates for auditing purposes.

The project demonstrates practical usage of:

* Database normalization
* Foreign keys and check constraints
* Stored procedures
* ACID transactions
* Error handling using `TRY...CATCH`
* Automatic account number generation
* Trigger-based audit logging

# Core Design

## 1. Atomic Customer Onboarding

Creating a customer also creates a bank account.

Both operations execute inside a single transaction.

```sql
BEGIN TRANSACTION

EXEC Pr_C_addCustomer ...

EXEC Pr_A_addAccount ...

COMMIT
```

If any step fails, the transaction is rolled back so the database remains consistent.

---

## 2. Stored Procedure API

Core tables are intended to be modified through stored procedures instead of direct SQL statements.

Benefits include:

* centralized validation
* reusable business logic
* simpler application integration
* reduced duplication

---

## 3. Automatic Account Number Generation

Account numbers are generated automatically using the format

```
PK-{BranchID}-{Year}-{CustomerID}
```

Example

```
PK-1-2026-1
```

This keeps account numbers unique while also encoding useful information.

---

## 4. Audit Logging

An `AFTER UPDATE` trigger on `A_ACCOUNTS` records balance modifications.

Each audit record stores:

* previous value
* updated value
* timestamp
* SQL Server user
* affected table

This provides a history of balance changes without requiring changes to application code.

#Example Usage

## Create Customer and Account

```sql
EXEC Pr_C_A_createCustomerAndAccount
    'Ahmed',
    'Ali',
    '35201-1111111-1',
    '03001112223',
    'ahmed@gmail.com',
    1,
    1,
    'Current';
```

Verify

```sql
SELECT
    c.C_FirstName,
    a.A_AccountNumber,
    b.B_BranchName
FROM C_CUSTOMERS c
JOIN A_ACCOUNTS a
ON c.C_CustomerId = a.C_CustomerId
JOIN B_BRANCHES b
ON a.B_BranchId = b.B_BranchId;
```

---

## Transfer Funds

```sql
EXEC Pr_T_makeTransfer
    'PK-1-2026-1',
    'PK-2-2026-2',
    3000.00,
    'Rent Payment';
```

The procedure

* withdraws from the source account
* deposits into the destination account
* creates linked transaction records
* executes inside a transaction

---

EXEC Pr_T_makeDeposit

## Withdraw

#Testing

### Duplicate Customer

```sql
EXEC Pr_C_A_createCustomerAndAccount
    'Ahmed',
    'Ali',
    '35201-1111111-1',
    NULL,
    NULL,
    1,
    1,
    'Current';
```

Expected:

* Duplicate CNIC detected
* Transaction rolled back

---

### Overdraft Protection

```sql
EXEC Pr_T_makeWithdraw
    'PK-1-2026-1',
    100000,
    'Overdraft Test';
```

Expected:

* Withdrawal rejected
* Balance unchanged

---

### Audit Logging

```sql
UPDATE A_ACCOUNTS
SET A_Balance = 999999
WHERE C_CustomerId = 1;
```

Check

```sql
SELECT *
FROM AL_AUDITS_LOGS;
```

The trigger automatically records the update.
